Proposal: Alinear panel Blocked con /v1/agent/status (P1.1)

Objetivo
- El panel Blocked de la UI debe leer el estado actual desde el brain (proxy) para mostrar blocked y refrescar tras Apply/Reject.

Contexto
- Fuente de verdad: GET /v1/agent/status desde brain_server vía ui_proxy.
- Estado esperado (ejemplo): {
  "ok": true,
  "room_id": "<room>",
  "blocked": null | {
    "proposal_id": "...",
    "step_id": "...",
    "required_approve": true,
    "summary": "..."
  }
}
- Gating: Apply/Reject proxyean a /v1/agent/apply|reject; tras resolución, blocked debe pasar a null.

Cambios propuestos (backend UI Proxy)
1) Endpoint GET /ui/api/blocked
   - Llama al brain: GET /v1/agent/status (propagando room_id actual; desde query/cookie/context según implementación existente).
   - Responde solo { blocked, ts, room_id } para simplicidad del panel.
   - Maneja errores: si brain no responde, retornar { blocked: null, error: "unreachable" } y loggear.

2) Endpoints POST /ui/api/apply y /ui/api/reject
   - Ya existen según ROADMAP; asegurar que tras el 200/ok se fuerce un refresh del panel (frontend) contra /ui/api/blocked.
   - Devolver estructura: { ok: true, resolved: true, proposal_id }.

Cambios propuestos (frontend)
- Panel Blocked:
  - Al montar, fetch /ui/api/blocked y renderizar:
    - Si blocked == null -> "Blocked: —" (guion largo o ndash según style).
    - Si blocked != null -> mostrar step_id/proposal_id y botones Apply/Reject.
  - Al hacer click en Apply/Reject:
    - Deshabilitar botones mientras se envía.
    - Llamar POST /ui/api/apply|reject con proposal_id.
    - Tras respuesta ok, refrescar /ui/api/blocked y re-renderizar.
  - Auto-refresh opcional cada 2s mientras existan propuestas activas.

Contratos/datos
- GET /v1/agent/status: usar room_id coherente con SSOT.
- blocked desaparece tras apply exitoso (expected behavior); la UI debe reflejarlo sin requerir recarga manual.

Tests (smoke)
1) Positivo:
   - Crear plan con un paso write/append (requiere aprobación).
   - Run -> aparece blocked con proposal_id/step_id.
   - En UI: panel muestra botones; click Apply -> en <=2s panel pasa a "Blocked: —".
   - Verificar el artifact (archivo modificado) y /status -> blocked=null.
2) Negativo:
   - Si no hay blocked y se hace click Apply/Reject: no acción, UI muestra notificación y mantiene "Blocked: —".
3) Robustez:
   - Si /v1/agent/status falla, panel muestra estado degradado y sugiere Refresh.

Notas de implementación
- Conservar coherencia de room_id en todas las llamadas.
- Loggear eventos en audit.ndjson a través del brain (no escribir audit desde UI directamente).
- No introducir dependencias nuevas sin revisión.

Riesgos y mitigaciones
- Riesgo: UI cachea estado antiguo -> mitigar con no-cache en /ui/api/blocked y refresco tras Apply/Reject.
- Riesgo: Cambios de schema -> validar presence/shape de blocked y tolerar campos extra.

Definición de hecho
- Cuando status.blocked existe, se ven botones Apply/Reject.
- Tras Apply/Reject exitoso, en <=2s el panel muestra "Blocked: —" y /status refleja blocked=null.
- Rechazos reflejan estado final coherente en plan.
